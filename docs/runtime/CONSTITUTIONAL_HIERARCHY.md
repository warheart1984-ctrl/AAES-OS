# Constitutional Hierarchy and Independence Standard

Status: normative invariant

```text
Constitution
  → Standards
    → Specifications
      → Constitutional Conformance Suite (CCS)
        → Reference Runtime (CRK-1 / AAES-OS)
          → Certified Implementations
```

The arrow denotes derivation and evaluation order, not software dependency. The Constitution is authoritative. Standards derive obligations. Specifications make them testable. CCS evaluates specifications independently. CRK-1 is a reference citizen, and AAES-OS is a non-normative execution substrate. Certified implementations are independent constitutional subjects; they do not inherit validity from CRK-1.

CCS test vectors, expected outcomes, schemas, and certification rules must be runnable without importing reference-runtime implementation code. A reference-runtime result is evidence, never the expected answer merely because CRK-1 produced it. Certification requires a declared implementation/version, pinned standards/specifications, reproducible CCS execution, receipts, and reviewer identity.

## Corpus neutrality

The runtime accepts registered constitutional benchmark corpora without embedding corpus-specific authority into the core lifecycle. RBOC is Constitutional Benchmark Corpus 1, not the Constitution. Historical, enterprise, scientific, healthcare, finance, public-administration, and AI-governance corpora use the same source/evidence/provenance/admission rules. A corpus profile may narrow handling for jurisdiction, privacy, risk, or evidence quality; it may not override constitutional invariants.
